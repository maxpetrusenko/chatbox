Flow

Proposal and challenge loop

The loop repeats for every stage. Stage changes do not change the protocol.

human intent
  ↓
shared memory updated
  ↓
one agent proposes
  ↓
other agent challenges and improves
  ↓
decision locks best packet
  ↓
execution runs only from locked packet
  ↓
proof ledger updated
  ↓
drift heartbeat runs
  ↓
stage promotion or repair
Default rule

Agreement is not the first goal. The first goal is a stronger alternative. Only after a better option is challenged and either accepted or rejected should the packet lock.

Memory

Three memory surfaces

Claude and Codex keep their own working memory while shared state carries the canonical truth.

  • risks

Current stage, locked packet, disagreement ledger, proof ledger, drift flags, and reply lock.

Heartbeats

Two heartbeats keep the loop honest

One reports coordination state. One reports freshness and mismatch.

Coordination heartbeat

  • active stage
  • proposal owner
  • challenge owner
  • blocked or unblocked
  • next move

Drift heartbeat

  • artifact versus repo
  • design versus implementation
  • plan versus current code
  • review claim versus proof
  • stale edges

Human skim contract

  • show stage
  • show best current packet
  • show open disagreement
  • show proof status
  • show drift status
How To Use It

Operator path

  • Capture one plain language intent.
  • Create or update the intent chain.
  • Write active stage and reply lock to shared state.
  • Let one agent propose the next packet.
  • Force the other to challenge and improve it.
  • Lock the stronger packet.
  • Execute from the locked packet only.
  • Record proof and run drift heartbeat.
