Case Study Analysis

TutorMeAI's core challenge is turning a chat product into a platform — one where third-party apps run inside the conversation while preserving the safety guarantees that 10,000 school districts depend on daily. The moment outside code renders UI and exchanges data with a chatbot used by children, every design decision becomes a trust decision.

The first key problem is the trust boundary between the platform and third-party apps. A third-party app with access to the parent page's DOM, chat history, or student data represents the default risk of embedding foreign code. The trade-off is between developer convenience and student safety. Full host access makes integration trivial — and abuse equally trivial. We chose sandboxed iframes with origin-checked postMessage as the communication bridge. Apps operate within their own frame. They receive only the scopes approved for them, and all state they want the chatbot to understand must be explicitly published through a structured event contract. This requires more work from app developers, but it means a broken or hostile app stays contained within its sandbox.

The second problem is communication and state. The chatbot must stay aware of what is happening inside an app — a chess board mid-game, a playlist being built, a weather result on screen — through structured data rather than pixel-level inference. We decided that the app must publish structured state snapshots back to the platform after each meaningful interaction. Those snapshots become part of the conversation context, so when a student asks "what should I do here?" the model answers from real data. The trade-off is that apps carry the responsibility of deciding what state to share, which adds complexity to the developer contract. But the alternative — inferring state from rendered UI — is fragile, unreliable, and impractical at scale.

The third problem is authentication across trust boundaries. Some apps require user-level credentials, like a Spotify integration that creates playlists on behalf of a student. Letting the embedded app run its own OAuth flow inside an iframe introduces phishing risk, malicious redirects, and insecure token storage. We chose platform-mediated authentication: the chat platform handles OAuth consent, stores and refreshes tokens, and passes only scoped capabilities to the app at invocation time. The ethical weight here is significant — these are children's accounts. The platform must be the single point of credential trust.

The ethical dimension runs through every layer. Content moderation for apps used by minors is a first-class concern. We chose a reviewed allowlist model rather than an open marketplace — every app must be approved before appearing in any classroom. App descriptions and capabilities are moderated. Emergency kill switches let administrators disable an app instantly across all districts. This is more restrictive than an open ecosystem, but for K-12 education the cost of a harmful app reaching students far exceeds the cost of slower developer onboarding.

What we landed on is a manifest-based plugin architecture with sandboxed rendering, structured event communication, platform-owned auth, and an admin-reviewed app registry. It prioritizes defensibility over openness — the right posture for a product whose users are children and whose customers are school districts that expect zero safety incidents.

Recommended architecture

System design

Messaging and history already exist in the product. This section covers the missing requirements only: app context, multi-turn continuity, tool registration, tool schemas, embedded UI, structured invocation, completion signaling, and independent app state.

Idea 1 · Fastest

Local plugin contract

  • Apps register capabilities directly with the host product.
  • Tool schemas live in the same platform runtime, so discovery is simple.
  • UI renders as trusted internal components inside the chat area.
  • Multi-turn context is easiest because app state and chat state share one store.
  • Focus: optimized for speed and a tightly coupled host boundary.

Best for: fastest prototype, lowest realism.

Idea 2 · Recommended

Manifest + iframe bridge

  • Each app registers through a manifest: id, capabilities, tool schemas, widget URL, auth type, allowed origins.
  • The model discovers tools from manifest-derived schemas and invokes them with structured parameters.
  • The UI renders inside the chat as a sandboxed iframe, so the app can own its own interface and state.
  • Multi-turn works through state snapshots sent back to the platform after each meaningful user action.
  • The app signals complete, error, and state.update, so the chat stays aware while the app retains its internals.

Best for: strongest fit to the assignment.

Idea 3 · Most extensible

Remote tool server + widget shell

  • Apps expose capabilities through a remote tool server, while UI comes from a separate widget endpoint.
  • Tool discovery is clean and protocol-shaped, good for future ecosystem growth.
  • Multi-turn context still depends on structured state echo from the app back to the platform.
  • Completion and error signals are explicit lifecycle events driven by the app contract.
  • Weak point: bigger build cost and more integration pieces this week.

Best for: long-term platform story, highest complexity.

Web search finding 1

Best UI boundary

OpenAI Apps SDK patterns, MDN iframe sandbox guidance, and OWASP third-party JS guidance all point toward isolated embedded UI with a dedicated host boundary.

Why: lets the app render rich UI in chat while keeping the host product in control.

Web search finding 2

Best communication model

MDN postMessage plus OAuth browser-app guidance support a narrow, origin-checked message bridge with explicit event types.

Why: the chatbot gets structured app state instead of guessing from visible UI.

Web search finding 3

Best auth model

RFC 9700 and browser-app OAuth guidance favor platform-mediated auth with PKCE and token handling outside the widget.

Why: authenticated apps connect safely with platform-held long-lived secrets and scoped session access.

Signal chain

User asks for task
  → model selects tool from registered schemas
    → platform validates app + scope + auth
      → widget mounts inline in chat
        → structured invoke sent to app
          → app emits state.update snapshots
            → snapshots appended to chat context
              → user asks follow-up
                → model answers from chat + app state
                  → app emits complete or error
                    → chat continues naturally

Key rule: published state becomes model context through the structured bridge.

Recommended contract

Register
App submits a manifest with capabilities, auth pattern, widget endpoint, and discoverable tool schemas.
Discover
The platform turns those schemas into callable tools the chatbot can choose during a conversation.
Render
When selected, the app UI mounts inline in the chat experience as part of the same conversation surface.
Invoke
The chatbot sends structured parameters to the app through explicit typed inputs.
Signal
The app returns lifecycle events: state.update, complete, and error.
State
The app keeps its own internal state, while the platform stores summarized state needed for future turns.

How multi-turn context works

Turn 1
User asks for an app task. The chatbot chooses a tool and mounts the matching app UI.
Turn 2
User interacts with the app. The app publishes a structured snapshot such as board state, playlist draft, or selected options.
Turn 3
User asks a follow-up like “what should I do here?” The chatbot answers using chat history plus the latest app snapshot.
Turn 4
The app emits completion. The chatbot treats the result as conversation context and continues naturally after the tool finishes.
Rule
The platform uses published app state as the source of truth for chatbot understanding.

OAuth flow for authenticated apps

User clicks "Connect Spotify"
  → platform opens OAuth consent in popup (owned origin)
    → user approves scopes on provider site
      → provider redirects to platform callback
        → platform stores encrypted refresh token
          → platform mints scoped session token
            → session token passed to app via bridge
              → app calls provider API with session token
                → platform auto-refreshes on expiry

The app widget never handles raw refresh tokens or redirect flows.

Reference implementation

Chess: end-to-end sequence

Full lifecycle

1. User: "let's play chess"
2. Model: selects chess.start_game tool
3. Platform: validates chess app → approved, internal, scope ok
4. Platform: mounts chess widget iframe inline in chat
5. Platform: sends invoke { tool: "start_game", params: { color: "white" } }
6. Chess app: initializes board, emits state.update
   → { fen: "rnbqkbnr/...", turn: "w", move_count: 0 }
7. Platform: appends snapshot to conversation context
8. User: drags e2 → e4 on board UI
9. Chess app: validates move, updates board, emits state.update
   → { fen: "rnbqkbnr/.../P.../...", turn: "b", last_move: "e2e4" }
10. User: "what should I do here?"
11. Model: reads chat + latest chess snapshot → suggests strategy
12. User: plays several more moves via board UI
13. Chess app: detects checkmate, emits complete
    → { result: "checkmate", winner: "white", moves: 24 }
14. Platform: closes widget, adds result to conversation context
15. Model: "Great game! You won in 24 moves. Want to analyze it?"

Every step maps to a specific contract primitive: invoke, state.update, complete.

Requirements to product boundary

How it connects to the app

Only the required connection points: how third-party apps plug into the chat product and what crosses the boundary.

1. Chat surface

The app keeps its existing chat experience, but adds one new surface: an embedded app panel inside the conversation.

When the assistant selects a tool, the panel opens inline as part of the same conversation surface.

The chat transcript also shows app lifecycle messages: launched, waiting, connected, completed, and recovery states.

2. App host boundary

The host product loads the third-party UI in a sandboxed iframe and passes only minimal startup data: app id, session id, theme, and approved scopes.

All communication goes through a narrow event bridge: invoke tool, send state snapshot, emit completion, emit error.

The embedded app operates through a scoped bridge into the broader chat UI and approved platform APIs.

3. Platform services

The app needs a registry service for app manifests, a policy layer for safety checks, and a state service that stores app context alongside conversation context.

That is what lets the assistant answer follow-up questions like “what should I do here?” after the user has already interacted with the widget.

The model reads structured state published back through the contract as the primary app-context channel.

4. Auth and trust path

For authenticated apps, the user authorizes through the platform in a platform-owned consent flow.

The platform stores and refreshes tokens, then gives the app only the minimum approved capability needed for that session.

This is the key connection between the requirements and the app: the chat remains the trusted control plane, while third-party apps stay isolated workers.

Adjacent problems

Risks that still need answers

These sit next to the core plugin problem and shape whether districts trust the platform enough to buy and expand it.

01

District policy controls

Each district will want its own app allowlist, scope rules, moderation posture, and emergency disable controls.

02

Student privacy records

App snapshots, auth events, and tool outputs may become part of regulated student records and need retention rules.

03

Prompt injection path

Tool descriptions, app metadata, and state payloads can shape model behavior and need validation plus policy filtering.

04

Embed compatibility

Many third-party products ship CSP and frame policies that shape which apps qualify for the platform contract.

05

Accessibility quality

Embedded apps need keyboard support, readable semantics, and classroom-grade accessibility to fit district procurement.

06

Cost and observability

State snapshots, repeated tool loops, and multi-turn analysis increase model cost and raise the need for strong tracing.
