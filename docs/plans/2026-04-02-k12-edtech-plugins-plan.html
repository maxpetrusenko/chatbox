API Key Model

Who owns the key? Who pays? Who sees it?

Recommended

District-Owned Keys

District admin enters API keys once. All schools under the district share them. Teachers never see raw keys.

  • District pays for API usage (consolidated billing)
  • One key per service per district
  • School admin can override with own key if needed
  • Usage tracked per school for chargeback
  • Key rotation handled at district level
simple secure Canvas/Google pattern

Platform Default Keys

For free APIs (Google Maps Embed, PhET, GeoGebra), the platform ships with built-in keys. Zero setup.

  • Platform absorbs cost for free-tier APIs
  • Rate-limited per district to prevent abuse
  • District can upgrade to own key for higher limits
  • Fallback: if no district key, use platform key
zero friction free APIs only

Per-User Tracking: Platform Proxy

District key authenticates with the API. But how do we know which student made each call? The platform proxies every request and tags it with user context.

Usage Dashboard (Admin View)

  • Total calls this month per plugin (bar chart)
  • Per-school breakdown (table)
  • Per-student top users (for quota management)
  • Alert: "Wolfram Alpha at 80% monthly quota"
  • Cost estimate for paid-tier APIs
  • Export CSV for district reporting

Why Not Per-Student Keys?

  • Massive setup friction (teacher helps every student?)
  • Most APIs don't support sub-keys or key-per-user
  • FERPA doesn't require per-student keys, just audit trails
  • Platform proxy gives same visibility with zero student setup
  • Kids lose keys. Teachers lose time. Proxy just works.

Architecture

How new plugins fit into the existing ChatBridge plugin system.

Implementation Phases

Build order. Start with zero-config plugins (fastest), then API-key plugins, then OAuth.

Phase 1: Zero Config GeoGebra + PhET + Scratch + Internet Archive. No keys, no auth. Just manifests + UI HTML. 4 plugins, fastest wins.
Phase 2: API Key Infra pluginApiKeyStore + setup UI + new auth type 'api-key'. Then: Google Maps + Desmos. Two plugins that prove the key flow.
Phase 3: REST API Plugins Wolfram Alpha + Merriam-Webster. No iframe. Main process calls REST API, returns data to chat. New pattern.
Phase 4: OAuth Plugin Flat.io. Uses existing auth broker. App ID as api-key + OAuth for teacher editing. Proves combined flow.

Decisions Needed

Resolved Decisions

  1. Per-user tracking: Platform proxy pattern. District key for API auth, userId tagged on every call. No per-student keys. FERPA audit via proxy logs.
  2. API key ownership: District-owned keys. Encrypted in pluginApiKeyStore. School override optional. Platform defaults for free APIs.
  3. Rate limiting: Per-student limits (configurable). Quota alerts at 80%. Prevents abuse without per-student keys.

Still Open

  1. Plugin count: All 9 plugins? Or start smaller (GeoGebra + PhET + Google Maps + Wolfram = 4)?
  2. REST-only plugins (Wolfram, MW): Show results as formatted chat cards? Or build a simple result viewer widget?
  3. GeoGebra JS API vs iframe: JS API = AI full control. Pure iframe = simpler but display-only. Recommend JS API.
  4. Phase order: Phase 1-4 as shown, or reprioritize?
