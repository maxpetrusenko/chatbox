K12 Vision

What the platform should look like for a district deploying to schools.

Tenant Model

Hierarchical multi-tenant structure for K-12 deployment.

District Top-level tenant. Owns global policies, plugin allowlist/denylist, compliance settings. One district = one deployment.
School Belongs to a district. Can narrow (not widen) district catalog. Has own admin and approval queue.
Class / Group Belongs to a school. Teacher assigns active plugins per class. Students see only class-scoped plugins.
User Teacher or Student. Role determines plugin install/use/admin capabilities.

Data Model (simplified)

District {
  id, name, settings: PolicyConfig
  allowedPlugins: string[]       // global allowlist
  blockedPlugins: string[]       // global denylist
}

School {
  id, districtId, name
  pluginOverrides: { pluginId: 'allow' | 'block' }[]
}

Class {
  id, schoolId, teacherId, name
  activePlugins: string[]        // teacher-curated subset
}

User {
  id, email, role: 'district-admin' | 'school-admin' | 'teacher' | 'student'
  schoolId, districtId
}

PluginInstall {
  id, pluginId, schoolId
  status: 'pending' | 'ai-review' | 'approved' | 'rejected' | 'quarantined'
  requestedBy: userId
  reviewedBy: userId | 'ai-auto'
  safetyScore: number            // 0-100, AI-assigned
  installedAt: timestamp
}

Safety Pipeline

Default deny. Only signed/approved plugins reach students.

Layer 1: Static Manifest schema validation, CSP enforcement, sandbox attribute check, permission scope audit.
Layer 2: AI Pre-screen LLM reviews plugin code/content for age-inappropriate material, unsafe patterns, data exfiltration.
Layer 3: Sandbox Test Quarantined execution in isolated iframe. Monitor network calls, DOM access attempts, postMessage patterns.
Layer 4: Human Review Admin sees AI safety score + findings. Final approve/reject. Auto-approve if AI score > 90 and no flags.
Layer 5: Runtime Continuous monitoring. Rate limiting, network scope enforcement, output content filtering, usage logging.

Content Safety APIs

  • OpenAI Moderation API (free, fast)
  • Google Perspective API (toxicity scoring)
  • Azure Content Safety (age-graded)
  • Custom LLM classifier for K12 context

Plugin Sandbox Rules

  • sandbox="allow-scripts" only
  • No allow-same-origin for untrusted
  • CSP: declared domains only
  • postMessage origin + nonce validation
  • 10s handshake timeout

COPPA / FERPA

  • No PII collection without district consent
  • Plugin data stays within tenant boundary
  • Student data never sent to plugin servers
  • Auth tokens scoped, never shared cross-plugin
  • Audit trail for all data access

Audit Layer

Scheduled AI review of plugin usage and behavior. Central authority sees everything.

What AI Scans For

  • Unsafe prompts from students to plugin tools
  • Plugin outputs containing inappropriate content
  • Data exfiltration patterns (PII leaving tenant)
  • Abnormal usage spikes (abuse indicators)
  • Network calls to undeclared domains
  • Age-inappropriate content generation

Central Authority Powers

  • See all schools' plugin usage in one dashboard
  • Force-remove a plugin across all schools instantly
  • Pin plugin versions to prevent untested updates
  • Set district-wide content policies
  • Review and override AI safety decisions
  • Export audit reports for compliance

Legal Requirement

COPPA 2025 Final Rule (effective Apr 22, 2026)

  • Default is now opt-in, not opt-out
  • Third-party data sharing requires separate verifiable parental consent
  • Penalty: $51,744 per affected child per violation
  • Persistent identifiers (device IDs, cookies) trigger COPPA scope
  • Each plugin must declare data collection profile in manifest
Compliance

FERPA "School Official" Provision

  • District signs DPA designating vendor as school official
  • No per-parent consent needed IF DPA covers data use
  • Vendor cannot re-disclose or use for commercial purposes
  • District maintains control + deletion rights
  • Plugins outside DPA umbrella still need individual consent

Content Safety API Pipeline

Layered approach: fast + free first, detailed second, custom third.

Layer 1: OpenAI Mod Free, ~50ms. Categories: sexual, sexual/minors, harassment, hate, violence, self-harm, illicit. Binary flagged/not. First-pass filter.
Layer 2: Azure Safety Severity 0-6. Four categories (Hate, SelfHarm, Sexual, Violence). Severity >= 4: block. Severity 2-3: flag for human review. Age-based threshold tuning.
Layer 3: Prompt Shields Azure jailbreak detection. Catches injection attacks + indirect prompt injection. Block + increment abuse score per student account.
Layer 4: Custom Blocklist School/district-specific terms. Azure Blocklist API. Layered on top of base classifiers. District admin manages.
Layer 5: LLM Output Scan Same pipeline on responses. Run Layers 1-4 on every LLM response + plugin output before streaming to student.

Abuse scoring: Rolling 24h counter per student. 5+ flags = temp restrict account + notify teacher. Prevents automated abuse.

Security

Plugin Signing Model (Grafana pattern)

MANIFEST.txt {
  "pluginId": "com.vendor.name",
  "signatureType": "community|verified|district",
  "files": {
    "index.js": "sha256:abc123...",
    "manifest.json": "sha256:def456..."
  }
}

Levels:
  community  = self-signed, district opt-in only
  verified   = platform-reviewed, default allowed
  district   = private to one district's key

At install: verify signature with platform public key, SHA256 each file against manifest. Mismatch = reject.
