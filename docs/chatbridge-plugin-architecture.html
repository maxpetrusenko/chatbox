Plugin System

How ChatBridge registers third-party apps, defines tool schemas, embeds UI, invokes tools, signals completion, and manages independent state.

Context

What we're building and why

ChatBridge already has a working chat system (Vercel AI SDK streaming, TanStack Query session persistence, Zustand + Jotai state) and a standalone chess route. The missing piece is the programmatic interface that lets third-party apps plug into the chat experience — registration, tool invocation, inline UI, completion signaling, and independent state.

This document covers the concrete architecture for that plugin system, grounded in research across ChatGPT plugins, Anthropic MCP, Slack apps, VS Code extensions, and Figma plugins. Every pattern is mapped to our actual Electron + React + TanStack Router codebase.

Current architecture snapshot: Messages flow through submitNewUserMessage → generate → streamText which already supports tool injection via ToolSet (MCP tools, web search, knowledge base). The plugin system hooks into this existing tool pipeline.

Electron drag-and-drop handler 
// In renderer — listen for file drops on the app window
document.addEventListener('drop', async (e) => {
  e.preventDefault();
  const file = [...e.dataTransfer.files].find(f => f.name.endsWith('.cbplugin'));
  if (!file) return;

  const arrayBuffer = await file.arrayBuffer();
  const result = await window.electronAPI.installPlugin(arrayBuffer, file.name);

  if (result.success) {
    pluginRegistry.refresh();
    showToast(`Installed ${result.manifest.name}`);
  } else {
    showToast(`Failed: ${result.error}`, 'error');
  }
});

// In main process — IPC handler
ipcMain.handle('install-plugin', async (_event, buffer, filename) => {
  const zip = new AdmZip(Buffer.from(buffer));
  const manifestEntry = zip.getEntry('plugin.json');
  if (!manifestEntry) return { success: false, error: 'No plugin.json found' };

  const manifest = JSON.parse(manifestEntry.getData().toString('utf8'));
  if (!isValidManifest(manifest)) return { success: false, error: 'Invalid manifest' };

  // Check no entries escape plugin directory
  for (const entry of zip.getEntries()) {
    const resolved = path.resolve(pluginsDir, manifest.id, entry.entryName);
    if (!resolved.startsWith(path.resolve(pluginsDir, manifest.id))) {
      return { success: false, error: 'Path traversal detected' };
    }
  }

  const installPath = path.join(pluginsDir, manifest.id);
  zip.extractAllTo(installPath, true);
  return { success: true, manifest };
});
Custom protocol registration for plugin:// URLs 
// In main process — register before app.whenReady()
import { protocol } from 'electron';

protocol.registerSchemesAsPrivileged([
  { scheme: 'plugin', privileges: { standard: true, secure: true, supportFetchAPI: true } }
]);

app.whenReady().then(() => {
  protocol.registerFileProtocol('plugin', (request, callback) => {
    const url = new URL(request.url);
    const pluginId = url.hostname;                    // plugin://chess/dist/ui.html
    const filePath = decodeURIComponent(url.pathname); // /dist/ui.html

    const plugin = pluginRegistry.get(pluginId);
    if (!plugin) return callback({ error: -6 });      // FILE_NOT_FOUND

    const safeBase = path.resolve(plugin.installPath);
    const resolved = path.resolve(safeBase, filePath.slice(1)); // remove leading /

    // Prevent path traversal
    if (!resolved.startsWith(safeBase)) return callback({ error: -10 }); // ACCESS_DENIED

    callback({ path: resolved });
  });
});

Tool name normalization (mirrors MCP controller pattern) 
// Manifest tool names stay local; ToolSet names become global.
function exposePluginToolName(pluginId: string, toolName: string) {
  const slug = pluginId.split('.').at(-1)?.replace(/[^a-zA-Z0-9_-]/g, '_').toLowerCase() ?? 'plugin';
  return `plugin__${slug}__${toolName}`;
  // "com.chatbridge.chess" + "new_game" => "plugin__chess__new_game"
}
Key insight from ChatGPT: description_for_model is a system prompt fragment. Write it like a prompt — when to use, when NOT to use, what it returns — not a docstring.

2 · Tool schema definition

How the chatbot discovers and invokes tools

MCP's inputSchema format internally, translated to provider-specific formats at call time.

Schema format decision

A
Format OpenAPI spec ChatGPT style. Heavyweight, requires HTTP endpoints per tool.
B ✓
Format MCP inputSchema JSON Schema per tool. Lightweight. Adapts to both OpenAI & Anthropic.
C
Format Custom DSL Non-standard. Every plugin author has to learn it. No ecosystem.

Provider comparison

OpenAI

Function calling

  • "strict": true → guaranteed compliance
  • Needs additionalProperties: false
  • Returns arguments as JSON string
Anthropic

Tool use

  • JSON Schema draft-07 compatible
  • Returns parsed input object
  • Results via tool_result block
MCP

JSON-RPC tools

  • Both input and output schemas
  • Annotations: readOnly, destructive
  • Dynamic tool list updates
Provider adapter code 
// src/shared/plugin-types.ts

interface PluginTool {
  name: string;            // local: "new_game"
  description: string;
  inputSchema: JSONSchema7;
}

interface RegisteredPluginTool extends PluginTool {
  pluginId: string;
  exposedName: string;     // global: "plugin__chess__new_game"
}

// Adapter: PluginTool → OpenAI
function toOpenAITool(tool: RegisteredPluginTool) {
  return { type: 'function' as const, function: {
    name: tool.exposedName, description: tool.description,
    strict: true, parameters: tool.inputSchema,
  }};
}

// Adapter: PluginTool → Anthropic
function toAnthropicTool(tool: RegisteredPluginTool) {
  return { name: tool.exposedName, description: tool.description,
    input_schema: tool.inputSchema };
}
ToolSet entry factory (how plugins enter the existing pipeline) 
function toToolSetEntry(plugin: PluginManifest, tool: RegisteredPluginTool) {
  return {
    description: `${plugin.description_for_model}\n\nTool: ${tool.description}`,
    inputSchema: tool.inputSchema,
    execute: async (args: unknown) => {
      const instanceId = pluginRuntime.activate({ pluginId: plugin.id, toolName: tool.exposedName, args });
      return { status: 'accepted', pluginId: plugin.id, instanceId, summary: `${plugin.name} opened.`, keepAlive: true };
    },
  };
}
Integration point: Plugin tools join the existing ToolSet in stream-text.ts:296-313 alongside MCP tools, web search, and knowledge base. No separate execution path.

3 · Rendering plugin UI in chat

Sandboxed iframes inside the conversation

When the LLM invokes a plugin tool, the platform renders the plugin's UI as a sandboxed iframe inline in the chat message stream.

Embedding mechanism decision

A
Mechanism Electron webview tag Deprecated by Electron team. Unstable. Security risk.
B ✓
Mechanism Sandboxed <iframe> Standard web. Lives in React DOM. postMessage bridge.
C
Mechanism WebContentsView Main process OS overlay. Can't participate in React layout.

Sandbox attributes

AttributeGrantsDecision
allow-scriptsJS executionYes
allow-same-originOrigin, cookies, storageNo — state goes through host bridge
allow-formsForm submissionNo
allow-top-navigationNavigate parentNo
allow-popupswindow.open()No
Design consequence: allow-scripts without allow-same-origin → opaque origin. The plugin has zero ambient storage. All durable state must flow through the host postMessage bridge. This is a feature, not a bug.

4 · Bidirectional communication

Host ↔ plugin message protocol

Typed postMessage protocol with handshake, nonce validation, rate limiting, and MessageChannel for RPC.

Message type definitions (src/shared/plugin-protocol.ts) 
// Host → Plugin
type HostToPluginMessage =
  | { type: 'PLUGIN_INIT'; instanceId: string; nonce: string;
      payload: { pluginId: string; sessionId: string; toolCallId: string; theme: 'light'|'dark'; initialState: unknown } }
  | { type: 'TOOL_INVOKE_PORT'; instanceId: string; nonce: string; callId: string; toolName: string; args: unknown }
  | { type: 'STATE_PUSH'; instanceId: string; nonce: string; state: unknown }
  | { type: 'THEME_CHANGE'; instanceId: string; nonce: string; theme: 'light' | 'dark' }
  | { type: 'LIFECYCLE'; instanceId: string; nonce: string; event: 'mount' | 'unmount' | 'focus' | 'blur' };

// Plugin → Host
type PluginToHostMessage =
  | { type: 'PLUGIN_READY'; instanceId: string; nonce: string; version: string }
  | { type: 'TOOL_RESULT'; instanceId: string; nonce: string; callId: string; result: unknown }
  | { type: 'TOOL_ERROR'; instanceId: string; nonce: string; callId: string; error: string }
  | { type: 'STATE_UPDATE'; instanceId: string; nonce: string; state: unknown }
  | { type: 'COMPLETION'; instanceId: string; nonce: string; payload: CompletionPayload }
  | { type: 'RESIZE_REQUEST'; instanceId: string; nonce: string; width: number; height: number };
Message handler with all security checks 
useEffect(() => {
  const handler = (event: MessageEvent) => {
    if (event.origin !== 'null') return;                     // opaque-origin sandbox
    if (event.source !== iframeRef.current?.contentWindow) return; // our iframe only
    if (!rateLimiter.current.consume()) return;               // 30 burst, 10/sec
    if (!isPluginToHostMessage(event.data)) return;           // type guard
    if (event.data.instanceId !== instanceId) return;         // instance binding
    if (event.data.nonce !== nonceRef.current) return;        // session nonce

    switch (event.data.type) {
      case 'PLUGIN_READY': /* send PLUGIN_INIT */ break;
      case 'TOOL_RESULT':  /* resolve pending tool call */ break;
      case 'STATE_UPDATE': /* persist plugin state */ break;
      case 'COMPLETION':   /* inject into chat context */ break;
    }
  };
  window.addEventListener('message', handler);
  return () => window.removeEventListener('message', handler);
}, []);
Why nonce: Opaque-origin iframes report event.origin === 'null', so origin pinning alone is insufficient. The per-instance nonce (passed via query param at iframe load) proves the message came from the iframe we mounted, not a rogue frame.

5 · Completion signaling

How a plugin tells the chat "I'm done"

The hardest problem. Short-lived tools complete immediately. Long-lived tools (chess) use a two-phase model.

Completion strategy decision

A
Strategy Synchronous block LLM waits for plugin to finish. Blocks conversation. Bad for games.
B ✓
Strategy Two-phase ACK/Complete Immediate ACK unblocks chat. Final completion when done. Best of both.
C
Strategy Polling Host polls plugin on every user message. Wasteful. Race-prone.
Phase 1 · Task started

Immediate ACK

  • Plugin sends TOOL_RESULT with initial state
  • LLM gets: "Chess game started. Board rendered."
  • Plugin UI stays mounted (keepAlive: true)
  • User can chat while plugin is active
Phase 2 · Task done

Final completion

  • Plugin sends COMPLETION with results
  • LLM gets: structured data (winner, PGN)
  • Plugin UI may unmount or stay for review
  • Context retained for follow-up questions
CompletionPayload structure 
interface CompletionPayload {
  summary: string;                  // "Game over. White won by checkmate in 32 moves."
  structuredData?: unknown;         // { winner: 'white', moves: 32, pgn: "1. e4 e5 ..." }
  continueConversation: boolean;    // true = LLM responds to result
  keepAlive: boolean;               // true = don't unmount plugin UI
  stateSnapshot?: unknown;          // full state for replay/review
}
Chess lifecycle: "let's play chess" → LLM calls plugin__chess__new_game → Phase 1 ACK, board mounts → user plays, asks "what should I do?" → LLM calls get_state + suggest_move mid-game → checkmate → Phase 2 COMPLETION with PGN → "how did I do?" → LLM has full context.

6 · Independent state management

How plugins own their state

Two-tier model. Opaque-origin iframes have zero ambient storage — all persistence goes through the host bridge.

State architecture decision

A
Approach allow-same-origin + IndexedDB Plugin manages own storage. Less secure. Cleanup burden on host.
B ✓
Approach Host-mediated persistence Plugin sends state via postMessage. Host stores in namespaced keys. Clean uninstall.
C
Approach No persistence State lost on every remount. Unusable for games.
TierWherePersistenceExample
Tier 1Inside iframeNone — lost on unmountHover, drag, animation state
Tier 2Host storeSurvives restartFEN, move history, difficulty
State namespacing & storage 
// Storage keys scoped by pluginId + sessionId + instanceId
function getStateKey(pluginId: string, sessionId?: string, instanceId?: string): string {
  if (instanceId && sessionId) return `plugin:${pluginId}::session:${sessionId}::instance:${instanceId}`;
  if (sessionId) return `plugin:${pluginId}::session:${sessionId}`;
  return `plugin:${pluginId}::global`;
}

// On uninstall: clean up all keys for this plugin
async function cleanupPluginState(pluginId: string) {
  const allKeys = await storage.keys();
  const pluginKeys = allKeys.filter(k => k.startsWith(`plugin:${pluginId}::`));
  await Promise.all(pluginKeys.map(k => storage.removeItem(k)));
}

7 · Error handling & resilience

What happens when things break

Every failure mode has a defined recovery path. The chat must never hang or crash because of a plugin.

FailureDetectionRecovery
iframe fails to load PLUGIN_READY not received within 10s of mount Show inline error card: "Plugin failed to load." Inject tool_result with is_error: true so LLM can explain to user.
Plugin crashes mid-game iframe onerror event, or heartbeat timeout (no message in 60s during active session) Show "Plugin stopped responding" overlay with Retry button. Persist last known state. On retry, remount iframe with saved state.
Tool call timeout MessageChannel response not received within 30s Reject the promise. Host returns TOOL_ERROR to LLM: "Tool timed out." LLM apologizes and may retry.
Malformed completion payload Type guard / schema validation fails on COMPLETION message Log warning. Ignore the message. Plugin stays mounted. No chat context corrupted.
Rate limit exceeded Token bucket exhausted (>30 burst or >10/sec sustained) Drop excess messages silently. Log warning. Plugin continues but may miss state updates.
Plugin sends during wrong lifecycle Messages arrive before PLUGIN_READY handshake or after COMPLETION Ignored. Type guard + state machine prevents acting on out-of-order messages.
Timeout & error boundary implementation 
// In PluginFrame.tsx
const READY_TIMEOUT_MS = 10_000;
const TOOL_TIMEOUT_MS = 30_000;

useEffect(() => {
  const timer = setTimeout(() => {
    if (!isReady) {
      setError('Plugin failed to load within 10 seconds.');
      onCompletion?.({
        summary: `Plugin "${pluginId}" failed to load.`,
        continueConversation: true, keepAlive: false,
      });
    }
  }, READY_TIMEOUT_MS);
  return () => clearTimeout(timer);
}, [isReady]);

</ErrorBoundary>
Design principle: A broken plugin must never break the chat. Every error path results in either a visible error card or an is_error: true tool result that the LLM can respond to gracefully.

The Plugin SDK

Plugin authors include a small SDK script. It wraps postMessage into a clean API inspired by VS Code's acquireVsCodeApi().

plugin-sdk.js (bundled into plugin's ui.html) 
class ChatBridgePlugin {
  #targetOrigin = '*';  // opaque-origin iframes must use '*'
  #instanceId;
  #nonce;
  #onInit;
  #onToolInvoke;

  constructor() {
    const params = new URLSearchParams(location.search);
    this.#instanceId = params.get('instanceId');
    this.#nonce = params.get('nonce');
    window.addEventListener('message', this.#handleMessage.bind(this));
  }

  // Call when your UI is ready
  ready(version = '1.0.0') {
    window.parent.postMessage(
      { type: 'PLUGIN_READY', instanceId: this.#instanceId, nonce: this.#nonce, version },
      this.#targetOrigin
    );
  }

  // Register handlers
  onInit(handler)       { this.#onInit = handler; }
  onToolInvoke(handler) { this.#onToolInvoke = handler; }

  // Send state update to host (debounce in your code)
  updateState(state) {
    window.parent.postMessage(
      { type: 'STATE_UPDATE', instanceId: this.#instanceId, nonce: this.#nonce, state },
      this.#targetOrigin
    );
  }

  // Signal task completion
  complete(payload) {
    window.parent.postMessage(
      { type: 'COMPLETION', instanceId: this.#instanceId, nonce: this.#nonce, payload },
      this.#targetOrigin
    );
  }

  #handleMessage(event) {
    const msg = event.data;
    if (msg?.instanceId !== this.#instanceId || msg?.nonce !== this.#nonce) return;
    if (msg.type === 'PLUGIN_INIT')       this.#onInit?.(msg.payload);
    if (msg.type === 'TOOL_INVOKE_PORT')  this.#onToolInvoke?.(msg, event.ports[0]);
  }
}

  <script>
    // Inline the SDK or import it
    const params = new URLSearchParams(location.search);
    const instanceId = params.get('instanceId');
    const nonce = params.get('nonce');
    let count = 0;

    window.addEventListener('message', (e) => {
      if (e.data?.instanceId !== instanceId || e.data?.nonce !== nonce) return;
      if (e.data.type === 'PLUGIN_INIT') {

    }

      window.parent.postMessage({
        type: 'COMPLETION', instanceId, nonce,
        payload: { summary: `Counter reached ${count}.`, structuredData: { count },
                   continueConversation: true, keepAlive: false }
      }, '*');
    }
  </script>
