Manifest, widget, postMessage, host-owned auth.
Then K12 policy on top.

This is the `chatbox` plugin contract as it exists in code today. The plugin API is not only a manifest and a widget bridge. It also carries app-auth modes, proxy and district-key config, K12 trust metadata, and runtime behavior that only becomes true once the host exposes the plugin to the model.

Reality Check

What makes this repo-specific

A plugin can be installed and still not be callable. `stream-text.ts` only injects plugin availability when the intent heuristic passes, then the registry still filters by K12 scope, app auth, plugin auth, and hidden builtin state.
Manifest Surface

What `plugin.json` carries in this repo

Core identity

id, name, version, description, category, tools, widget entrypoint

Host auth

oauth2-pkce, device-flow, api-key, plus app auth like Chatbox AI login or K12 login

Proxy + district config

district-key requirements, quotas, tracking pattern, and platform-owned usage recording

K12 compliance

trust level, COPPA scope, grade targets, PII profile, allowed domains, signature type

Runtime Contract

What actually happens when the model calls a plugin tool

Message Directions

Host and plugin only speak in these lanes

  • Host → plugin: PLUGIN_INIT, TOOL_INVOKE, AUTH_STATUS
  • Plugin → host: PLUGIN_READY, STATE_UPDATE, COMPLETION
  • Plugin → host errors and auth hooks: TOOL_RESULT, ERROR, AUTH_REQUEST
Auth Matrix

What auth means in Chatbox

  • chatbox-ai-login: app hidden until Chatbox AI session is connected
  • k12-login: app depends on the K12 session state
  • oauth2-pkce or device-flow: host owns the broker, widgets get short-lived tokens
  • api-key or district key: configuration can be admin-only, with teacher save-disabled fallback
