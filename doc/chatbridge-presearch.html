Appendix-driven planning

Three solutions per phase

Each phase gets three viable approaches. Option B is the recommended path for this repo and timeline.

Phase 1 · Define Your Constraints

Scale, budget, security, and team capacity decide whether the architecture should be mostly local, hybrid, or platform-grade.

Solution A · Demo-first

Bundle apps into the product

Fastest Low infra Weakest trust boundary
  • Assume trusted apps only, shipped with the desktop/web client.
  • Support three first-party style apps with no public marketplace yet.
  • Best when the real goal is a fast proof of interaction patterns.

Tradeoff: quick demo, but it avoids the hardest third-party safety problem.

Solution B · Recommended

Reviewed remote apps with sandboxed UI

Balanced Realistic Defendable
  • Assume a reviewed allowlist of remote apps, each on its own origin.
  • Support 10k districts by isolating app UI and limiting scopes up front.
  • Fits the one week sprint without pretending a full ecosystem is finished.

Tradeoff: more moving parts, but it solves the actual assignment boundary.

Solution C · Ecosystem-first

Open app marketplace from day one

Highest ambition Strong long-term story Highest risk
  • Assume public developer onboarding, signed manifests, app review queues, and policy engine.
  • Best if the sprint were about platform strategy instead of shipping a credible demo.
  • Too much operational surface for the current deadline.

Tradeoff: impressive roadmap, but dangerous scope creep this week.

Phase 2 · Architecture Discovery

This phase decides the contract: how apps register, render, receive calls, and publish state.

Solution A · Local plugin host

Registry + React components

  • Apps register a manifest and ship a local React component loaded inside the renderer.
  • Tool calls are internal function calls, no iframe bridge required.
  • Works well for chess and weather, less honest for remote third-party trust.

Use when: speed matters more than isolation.

Solution B · Recommended

Manifest registry + sandboxed iframe SDK

  • Each app publishes metadata, scopes, tool schemas, allowed origins, and widget URL.
  • The chat shell renders apps in sandboxed iframes and exchanges structured events over postMessage.
  • A backend-for-frontend handles OAuth token exchange and app-side credential brokering.

Use when: you want a real plugin boundary and still need to ship this week.

Solution C · Protocol-first

MCP-style remote tool servers + widget shell

  • Tools are exposed by remote servers via a protocol like MCP; UI is attached separately as a widget.
  • Strong long-term interoperability and cleaner discovery semantics.
  • Higher integration cost for a one week student sprint.

Use when: the main goal is platform elegance, not fastest completion.

Phase 3 · Post-Stack Refinement

Hardening decides whether the demo feels safe, resilient, and teachable to graders.

Solution A · Minimal hardening

Add retries, logs, and obvious guardrails

  • Focus on timeout handling, app crash recovery, and visible loading states.
  • Add a simple allowlist and manual app approval process.
  • Enough for a demo, but thin for a school deployment story.

Tradeoff: fastest polish, weakest long-term argument.

Solution B · Recommended

Zero-trust default for apps

  • Permission-scoped manifests, per-app origins, moderated descriptions, audit logs, and OAuth via BFF.
  • Conversation state is platform-owned; app state is app-owned but echoed through signed events.
  • Strong enough to justify safety claims without boiling the ocean.

Tradeoff: more implementation work, but directly answers the grader rubric.

Solution C · Platform-grade operations

Policy engine, anomaly detection, and governance

  • Add app reputation, automated scanning, kill switches, and richer admin review workflows.
  • Best for a production roadmap or follow-up milestone.
  • Likely overbuilt for this sprint unless a team is parallelizing aggressively.

Tradeoff: strongest story, biggest schedule risk.

Recommended architecture

System design

Messaging and history already exist in the product. This section covers the missing requirements only: app context, multi-turn continuity, tool registration, tool schemas, embedded UI, structured invocation, completion signaling, and independent app state.

Idea 1 · Fastest

Local plugin contract

  • Apps register capabilities directly with the host product.
  • Tool schemas live in the same platform runtime, so discovery is simple.
  • UI renders as trusted internal components inside the chat area.
  • Multi-turn context is easiest because app state and chat state share one store.
  • Weak point: this does not prove a real third-party boundary very well.

Best for: fastest prototype, lowest realism.

Idea 2 · Recommended

Manifest + iframe bridge

  • Each app registers through a manifest: id, capabilities, tool schemas, widget URL, auth type, allowed origins.
  • The model discovers tools from manifest-derived schemas and invokes them with structured parameters.
  • The UI renders inside the chat as a sandboxed iframe, so the app can own its own interface and state.
  • Multi-turn works through state snapshots sent back to the platform after each meaningful user action.
  • The app signals complete, error, and state.update, so the chat stays aware without owning app internals.

Best for: strongest fit to the assignment.

Idea 3 · Most extensible

Remote tool server + widget shell

  • Apps expose capabilities through a remote tool server, while UI comes from a separate widget endpoint.
  • Tool discovery is clean and protocol-shaped, good for future ecosystem growth.
  • Multi-turn context still depends on structured state echo from the app back to the platform.
  • Completion and error signals are explicit lifecycle events, not inferred from chat text.
  • Weak point: bigger build cost and more integration pieces this week.

Best for: long-term platform story, highest complexity.

Web search finding 1

Best UI boundary

OpenAI Apps SDK patterns, MDN iframe sandbox guidance, and OWASP third-party JS guidance all point toward isolated embedded UI, not direct host-DOM execution.

Why: lets the app render rich UI in chat while keeping the host product in control.

Web search finding 2

Best communication model

MDN postMessage plus OAuth browser-app guidance support a narrow, origin-checked message bridge with explicit event types.

Why: the chatbot gets structured app state instead of guessing from visible UI.

Web search finding 3

Best auth model

RFC 9700 and browser-app OAuth guidance favor platform-mediated auth with PKCE and token handling outside the widget.

Why: authenticated apps connect safely without letting the embedded app own long-lived secrets.

Architecture

Chat App
├── Conversation UI
├── Tool Router
├── App Registry
├── Widget Host (sandboxed iframe)
├── App Context Store
├── Policy Guard
└── OAuth Broker

Third-Party App
├── manifest.json
├── tool schemas
├── widget UI
└── internal state

Host owns trust, routing, and conversation context. App owns UI and its own working state.

Signal chain

User asks for task
  → model selects tool from registered schemas
    → platform validates app + scope + auth
      → widget mounts inline in chat
        → structured invoke sent to app
          → app emits state.update snapshots
            → snapshots appended to chat context
              → user asks follow-up
                → model answers from chat + app state
                  → app emits complete or error
                    → chat continues naturally

Key rule: no UI scraping. Only published state becomes model context.

Data model

conversation/
  {conversation_id}/
    messages/
    active_apps/
      {app_instance_id}/
        app_id
        status
        approved_scopes.json
        last_tool_call.json

app_registry/
  {app_id}/manifest.json
  {app_id}/policy.json

oauth_connections/
  {user_id}/{app_id}.json

Conversation holds only the app state needed for future turns. Full internal app state stays with the app.

What you are actually building

1. App registration contract
2. Tool schema discovery layer
3. Inline widget host in chat
4. Structured invoke/event bridge
5. App snapshot → model context path
6. Completion/error lifecycle
7. Platform-mediated auth for protected apps

That is the minimum technical slice needed to satisfy the missing requirements.

Recommended contract

Register
App submits a manifest with capabilities, auth pattern, widget endpoint, and discoverable tool schemas.
Discover
The platform turns those schemas into callable tools the chatbot can choose during a conversation.
Render
When selected, the app UI mounts inline in the chat experience, not in a detached tab or separate product.
Invoke
The chatbot sends structured parameters to the app, never loose text-only guesses.
Signal
The app returns lifecycle events: state.update, complete, and error.
State
The app keeps its own internal state, while the platform stores summarized state needed for future turns.

How multi-turn context works

Turn 1
User asks for an app task. The chatbot chooses a tool and mounts the matching app UI.
Turn 2
User interacts with the app. The app publishes a structured snapshot such as board state, playlist draft, or selected options.
Turn 3
User asks a follow-up like “what should I do here?” The chatbot answers using chat history plus the latest app snapshot.
Turn 4
The app emits completion. The chatbot treats the result as conversation context and continues naturally after the tool finishes.
Rule
The platform never assumes app state from pixels alone. The app must publish the state it wants the chatbot to understand.

Requirements to product boundary

How it connects to the app

Only the required connection points: how third-party apps plug into the chat product and what crosses the boundary.

1. Chat surface

The app keeps its existing chat experience, but adds one new surface: an embedded app panel inside the conversation.

When the assistant selects a tool, the panel opens inline as part of the same conversation, not as a separate product area.

The chat transcript also shows app lifecycle messages: launched, waiting, connected, completed, failed.

2. App host boundary

The host product loads the third-party UI in a sandboxed iframe and passes only minimal startup data: app id, session id, theme, and approved scopes.

All communication goes through a narrow event bridge: invoke tool, send state snapshot, emit completion, emit error.

The embedded app never gets direct control over the broader chat UI or unrestricted platform APIs.

3. Platform services

The app needs a registry service for app manifests, a policy layer for safety checks, and a state service that stores app context alongside conversation context.

That is what lets the assistant answer follow-up questions like “what should I do here?” after the user has already interacted with the widget.

The model does not infer app state from UI alone. It receives structured state published back through the contract.

4. Auth and trust path

For authenticated apps, the user authorizes through the platform, not inside an untrusted widget-controlled flow.

The platform stores and refreshes tokens, then gives the app only the minimum approved capability needed for that session.

This is the key connection between the requirements and the app: the chat remains the trusted control plane, while third-party apps stay isolated workers.

Web research

Research sources used

2024 to 2025 guidance used to justify sandboxing, messaging, tool contracts, and auth choices.

OpenAI Apps SDK Reference developers.openai.com/apps-sdk/reference

Useful for tool descriptors, widget metadata, dedicated component origins, and JSON-RPC style UI bridge patterns.

OpenAI Apps Security & Privacy developers.openai.com/apps-sdk/guides/security-privacy

Supports the case for sandboxed iframes, strict CSP, and limiting widget capabilities by default.

MDN postMessage developer.mozilla.org/.../Window/postMessage

Grounds secure cross-origin messaging. Critical point: always validate sender and target origins.

MDN iframe sandbox developer.mozilla.org/.../HTMLIFrameElement/sandbox

Supports app isolation by selectively granting only the smallest needed iframe capabilities.

MDN CSP directives developer.mozilla.org/.../CSP/frame-ancestors developer.mozilla.org/.../CSP/sandbox

Explains both who may embed a page and what a sandboxed page is allowed to do once embedded.

RFC 9700 + Browser Apps draft rfc-editor.org/rfc/rfc9700.html datatracker.ietf.org/.../draft-ietf-oauth-browser-based-apps

Supports PKCE, secure in-browser messaging checks, and the choice to keep tokens away from untrusted widget code.

Model Context Protocol tools spec modelcontextprotocol.io/specification/2025-06-18/server/tools

Useful reference for tool discovery semantics and structured tool metadata design.

OWASP third-party JS cheat sheet cheatsheetseries.owasp.org/.../Third_Party_Javascript_Management_Cheat_Sheet.html

Reinforces iframe sandboxing, CSP hardening, and the general danger of giving third-party code too much power.
